Controlled max-agent loops

Let agents run hard without losing the plot.

Suede Agent Teams is the skill for important work that can break when one loose agent guesses, edits the wrong files, skips review, or calls something done before the live route, build, or public claim is verified.

It gives Codex a delivery room: scout the target, split only safe lanes, make every lane own an output, run reviewers against the result, recover from failed checks, lock release truth, and leave evidence the next agent can use.

What changes
  • One scout maps the repo, docs, dirty files, routes, and no-touch boundaries before edits start.
  • Builder lanes work only where file ownership is clear.
  • Review lanes try to break the result from production, public-claim, release, mobile, and handoff angles.
  • Release verifier owns build, live/API readback, and truth before anyone says shipped.

The useful part is control.

Agent Teams is not a bigger prompt. It is a control system for agent work. It says who scouts, who builds, who reviews, which files are safe, what evidence counts, and what happens when a check fails.

Use it when the work touches public pages, shared components, auth, payments, rights, provenance, registry flows, royalties, agent commerce, app releases, deployments, or anything with user WIP nearby.

Skip it for small cuts.

Do not bring a team to a one-line copy edit or a simple command. The skill earns its keep when coordination prevents wasted passes, file collisions, false claims, or release drift.

The roster is practical.

1

Scout

Finds the exact repo, docs, branch, dirty files, routes, live surfaces, account boundaries, and risky files before a builder touches anything.

2

Planner

Turns the request into atomic tasks with file targets, dependencies, acceptance checks, and a visible done signal.

3

Builder lanes

Make narrow edits inside owned files or surfaces. If lanes collide, the orchestrator sequences them instead of letting them stomp each other.

4

Design and visibility review

Checks rendered quality, mobile behavior, accessibility basics, CTA pull, proof, AI readability, and first-screen clarity.

5

Code grade and review

Grades ship risk A-F and hunts for bugs, regressions, security holes, public-claim drift, missing tests, and weak release behavior.

6

Release verifier

Owns the build, deploy, live/API readback, App Store or iOS truth when relevant, and the final public claim boundary.

Pick the loop that matches the risk.

Linear delivery

Scout, plan, build, verify, review, ship. Use it when sequencing matters more than speed.

Safe parallel build

Split lanes only when write surfaces do not collide. Good for copy, design, SEO, and QA lanes with clean ownership.

Adversarial review

One lane asks how the work fails in production, public use, release review, abuse, accessibility, mobile, or handoff.

Consensus review

Two review lenses merge blockers, plausible risks, accepted caveats, fixes now, and follow-ups that should not block.

Release lock

Build, deploy, live/API readback, public copy truth, and handoff are checked before a completion claim leaves the thread.

Recovery replay

If a check fails, stop broad work, isolate the failing unit, patch only the gap, and rerun the failed check before widening again.

Prompts that make it move.

Major page pass

Use $suede-agent-teams. Scout the repo and page first, then split copy, design, visibility, and QA lanes only where files do not collide. End with a ship gate and evidence handoff.

Release-bound work

Use $suede-agent-teams for a release lock loop. Verify branch, dirty files, build, live/API behavior, public claims, and handoff before calling this shipped.

Recovery

Use $suede-agent-teams recovery loop. Freeze broad work, isolate the failing check, patch only that gap, rerun the failed check, then widen only after it passes.

Claim boundary.

Agent Teams organizes work. It does not guarantee correctness, run private services, bypass review, clear legal rights, approve payouts, write to registries, or prove a deployment is live without a current verification step.
