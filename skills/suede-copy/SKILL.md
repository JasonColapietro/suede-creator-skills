# Suede Copy

Use this skill to write Suede copy that sells the work without sounding like a
template. The voice is direct, specific, public-safe, and grounded in Suede's
world: creator ownership, programmable IP, rights, provenance, registry-backed
media, royalty routing, licensing readiness, and agent commerce.

## Before Writing

Read any local product or brand context before asking questions:

- `AGENTS.md`
- `README.md`
- `PASSPORT.md`
- `AI_HANDOFF.md`
- `PRODUCT.md`
- `DESIGN.md`
- product marketing or brand notes
- task-specific docs

If context is missing, ask only for what blocks accurate copy:

- page or doc type;
- primary reader;
- one action the reader should take;
- product or skill being offered;
- proof that is safe to claim;
- claims, pricing, partners, or metrics that are not approved;
- traffic source or publication surface.

## Core Rules

- Choose clarity over cleverness.
- Name the outcome, not the feature list.
- Use Suede words only when they clarify the offer.
- Keep one idea per section.
- Use active voice with a clear actor.
- Replace vague claims with concrete actions, artifacts, links, commands, or
  proof.
- Write buttons as actions: `Install the skill`, `Create a rights passport`,
  `Audit a release folder`, `Read the docs`, `Open the repo`.
- Do not invent stats, testimonials, partners, pricing, availability, legal
  clearance, payout claims, or registry writes.
- Do not reduce Suede to a generic AI music app.
- Do not use em dashes in public copy.

## Suede Voice

Use this register:

- confident, not breathless;
- technical enough for builders;
- clear enough for creators;
- polished, not corporate;
- specific, not cute;
- operator-grade, not brochure-grade.

Good Suede copy names what the reader controls:

- register a work;
- verify rights;
- route royalties;
- publish a claim;
- package a release folder;
- prepare licensing evidence;
- make a work readable to agents;
- compare provenance;
- ship a public skill page.

## Page And Docs Structure

For a page, README, or docs surface, build this spine:

1. **Hero:** one sentence that names the outcome.
2. **Subhead:** one or two sentences that add the audience, workflow, and proof.
3. **Primary CTA:** the action the reader can take now.
4. **Proof:** files, scripts, docs, screenshots, URLs, live routes, examples, or
   commands.
5. **How it works:** three or four steps, each with a verb and result.
6. **Safety:** what the workflow does not claim or do.
7. **FAQ:** direct answers for objections and search intent.
8. **Final CTA:** repeat the action with less friction.

For a small section, use only the pieces that fit.

## SEO And GitHub Copy

For GitHub repositories, skill docs, and Pages sites, treat SEO as the umbrella
for search, AEO, and AI EO. Include:

- a search-ready title under 60 characters when practical;
- a meta description under 160 characters when practical;
- repo description under GitHub's practical limit;
- 8-20 topic keywords if the repo surface supports them;
- a first paragraph that repeats the durable entity names naturally;
- answer-ready definitions, FAQ copy, and proof links that AI summaries can
  cite without inventing facts;
- links to install docs, skill manifests, scripts, references, examples, live
  Pages, and source;
- a safe public claim boundary.

Prefer durable keywords:

- Suede Creator Skills
- Suede Rights Passport
- Music Release Metadata Linter
- Suedify
- Suede Copy
- AI EO
- AEO
- answer engine optimization
- Codex skills
- Claude Code skills
- SKILL.md
- music rights
- creator rights
- release readiness
- provenance
- royalty splits
- licensing readiness
- programmable IP
- agent commerce
- GitHub Pages

Use keywords because they help the right reader find the page. Do not cram a
keyword where a human would notice.

## SEO Audit Mode

When the user asks for an SEO audit, SEO rewrite, search optimization, metadata
pass, AEO pass, AI EO pass, answer-engine optimization, page audit, or full
audit, inspect the actual page, repo, or docs source before writing
recommendations. If a live URL exists, verify the live page. If only files
exist, inspect the source and rendered HTML when practical.

Run the audit in these lanes:

1. **Technical access:** canonical URL, robots/indexability, sitemap entry,
   redirects, status code, crawl path, and duplicate URL risk.
2. **Search and answer intent:** one primary reader, one primary query/theme,
   one AI-answer-ready definition, one action the page should earn, and any
   likely keyword cannibalization.
3. **Metadata:** title, meta description, Open Graph, Twitter card, image alt,
   author/publisher, and durable entity names.
4. **Structure:** one H1, useful H2/H3 hierarchy, section order, FAQ fit,
   internal links, and descriptive anchor text.
5. **Schema:** JSON-LD validity, FAQ/source fit, and whether each schema type
   matches visible page content.
6. **Copy quality:** directness, proof, claim boundaries, CTA clarity, trust
   language, filler removal, and Suede vocabulary fit.
7. **Conversion:** primary CTA, secondary CTA, proof stack, next-step friction,
   and whether the page tells the reader what to do.
8. **Verification:** commands, URLs, browser checks, schema parse, sitemap
   parse, and link sweep needed to prove the fixes.

For a full audit, produce ranked findings:

```text
[HIGH|MEDIUM|LOW] Finding title
Location:
Issue:
Fix:
Suggested copy:
Verification:
```

Then provide:

```text
SEO title:
Meta description:
H1:
Subhead:
Primary CTA:
Secondary CTA:
FAQ additions:
Internal links to add:
Schema changes:
Claim boundaries:
Verification checklist:
Copy score:
Ship gate: ship | ship-with-caveats | hold
```

Do not present SEO advice as complete if you did not check the current page or
source. Say what was audited and what was not.

## Anti-Slop Pass

Before delivering, remove:

- throat-clearing openers;
- filler transitions;
- adverbs and softeners;
- passive voice;
- inanimate subjects doing human actions;
- binary setup lines;
- dramatic fragments;
- rhetorical questions that answer themselves;
- vague declarations;
- three-item cadence when two items work;
- exclamation points;
- em dashes.

Score the copy before handoff:

```text
Directness: /10
Rhythm: /10
Trust: /10
Specificity: /10
Density: /10
Total: /50
```

Revise below 40/50. For public launch copy, aim for 44/50 or higher.

## Ship Gate

Do not ship copy when:

- the primary action is unclear;
- the page promises a feature the product does not implement;
- proof is fake or unverified;
- the copy hides a legal, payment, privacy, or release caveat;
- the writing sounds generic enough to belong to any AI product.

End with the exact copy, not a long explanation of the copy.
